% ABSTRACT
% =====================================================================
\begin{abstract}
As LLM-powered agents become the dominant consumers of analytical data systems,
a fundamental mismatch emerges: OLAP databases were designed for human users who
craft careful queries, not for agents that issue hundreds of exploratory probes
per session. Liu \etal~\cite{liu2025sao} characterize agentic workloads by
four properties---\emph{scale}, \emph{heterogeneity}, \emph{redundancy}, and
\emph{steerability}---and call for redesigning data systems to be agent-first.

This work presents \sysname, a transparent middleware layer that sits between
LLM agents and OLAP databases.  \sysname introduces three mechanisms:
(1)~\emph{intent-based access control} that parses SQL ASTs to enforce policies
at the semantic level (\eg, allowing aggregations over sensitive columns while
blocking row-level access);
(2)~\emph{session-based query learning} that persists agent sessions as
embeddings and surfaces relevant past queries to new agents with similar
goals, enabling cross-task knowledge transfer; and
(3)~a curated \emph{explore endpoint} that provides schema metadata, column
semantics, access-level annotations, and sample queries, eliminating the
redundant schema-discovery queries that dominate agentic workloads.

\sysname is evaluated on a distributed ClickHouse cluster with two benchmark
tracks.
A \emph{security benchmark} (5~attack scenarios, 200~probes) shows that the
middleware blocks 67~additional attacks versus unprotected ClickHouse,
reducing PII leakage from 300 to~0 rows.
An \emph{agent-efficiency benchmark} (8~analytical tasks, 3~models, up to
10~sequential agents per task) demonstrates that the middleware reduces total
queries hitting the OLAP system by 74.5\% (200$\to$51) and eliminates all
95~schema-exploration queries.  An explore-endpoint ablation shows that
without the endpoint, task success drops from 100\% to 12.5\%, confirming
that the explore layer is the foundational enabler for policy-compliant agent
interactions.  Cross-task transfer experiments show that queries stored by
agents solving one task reduce queries for semantically related tasks by 90\%.
Agent capability matters: GPT-4o achieves 62.5\% accuracy in 1.1~queries on
average, GPT-4o-mini achieves 45.8\% in 3.5~queries, while Qwen-3.5 scores
25\% despite the same middleware support, indicating that agent-first
middleware amplifies model capability rather than leveling it.
\end{abstract}

\section{Introduction}
\label{sec:intro}

The interface between data systems and their consumers is undergoing a
structural shift. Where human analysts once formulated precise SQL queries
informed by domain expertise and schema knowledge, LLM-powered agents now
interact with databases through iterative, exploratory probe
sequences. These agentic workloads differ qualitatively from
traditional analytics: they exhibit high \emph{scale} (hundreds of queries per
task), substantial \emph{heterogeneity} (mixing metadata exploration with
analytical computation), pervasive \emph{redundancy} (repeating similar
schema-discovery patterns across sessions), and latent
\emph{steerability} (responding to richer feedback than simple error
messages)~\cite{liu2025sao}.

This shift exposes three critical gaps in current OLAP architectures.

\paragraph{Security.}
Traditional role-based access control (RBAC)~\cite{sandhu1996rbac} operates at
the table or column level: a principal either can or cannot access a column.
But agentic intent is more nuanced. An agent computing \texttt{AVG(salary)
GROUP BY department} poses fundamentally different privacy risk than one
executing \texttt{SELECT name, salary FROM employees}---yet both access the
same column. Current systems cannot distinguish between these intents, forcing
administrators into a binary choice between permitting all access (risking PII
leakage) or blocking the column entirely (crippling legitimate analytics).

\paragraph{Resource governance.}
Unlike human analysts who self-regulate query frequency, agent swarms can flood
an OLAP cluster with hundreds of concurrent requests---mixing expensive
analytical scans with speculative exploration probes.  Most OLAP systems offer
per-user resource governors (max memory, max threads), but these are static
limits designed for tens of concurrent users, not for dynamically managing
thousands of agents.  There is no concept of binding an agent to a principal
identity across sessions, capping its concurrent queries, or throttling its
request rate with backoff signals.  Without such governance, a single
misconfigured agent loop can degrade query latency for the entire cluster.

\paragraph{Efficiency.}
Prior work~\cite{liu2025sao} observes that 80--90\% of agent queries in
text-to-SQL benchmarks are redundant schema-exploration
probes---\texttt{DESCRIBE TABLE}, \texttt{SHOW COLUMNS}, and speculative
\texttt{SELECT *} queries whose sole purpose is to map the schema landscape.
These probes consume tokens, exhaust rate limits, and inflate latency, yet
yield information the data system already possesses. No mechanism exists to
proactively surface this metadata in a form agents can directly consume.

\medskip
This work presents \sysname, a transparent middleware layer that addresses these
gaps without modifying the underlying OLAP engine.  \sysname contributes four
mechanisms:

\begin{enumerate}[leftmargin=*]
  \item \textbf{Intent-based access control} (\S\ref{sec:access}): SQL AST
        parsing with \texttt{sqlglot}~\cite{sqlglot2023} classifies query
        intent and enforces policies at the semantic level---allowing
        aggregations over sensitive columns while blocking row-level access
        to the same data.

  \item \textbf{Session-based query learning} (\S\ref{sec:learning}):
        Agent sessions are embedded and persisted in a vector store,
        enabling cross-agent and cross-task knowledge transfer.  Queries
        validated by one agent are surfaced to future agents with
        semantically similar goals.

  \item \textbf{Curated explore endpoint} (\S\ref{sec:explore}):
        Access-controlled schema metadata enriched with column semantics,
        access-level annotations, and sample queries eliminates redundant
        exploration.  Critically, the explore endpoint serves dual
        purpose: schema discovery \emph{and} policy guidance, teaching
        agents how to formulate compliant queries rather than merely
        blocking non-compliant ones.

  \item \textbf{Agent resource governance} (\S\ref{sec:governance}):
        Each agent is bound to a single principal with per-agent rate
        limits, concurrent query caps, and full audit logging---regardless
        of how many sessions or connections it opens.  Rate-limited
        requests receive \texttt{Retry-After} headers, enabling
        well-behaved agents to back off gracefully rather than overloading
        the cluster.
\end{enumerate}

By design, the middleware architecture places a protective layer between
agents and the OLAP cluster.  Every query passes through access control,
rate limiting, and audit logging before reaching the database---shielding
the cluster from both malicious and accidental abuse.  \sysname deploys as
an OLAP-compatible HTTP proxy, requiring zero changes to existing agent
tooling.  It is evaluated on a distributed ClickHouse cluster across two
benchmark tracks:

\begin{itemize}[leftmargin=*]
  \item A \emph{security benchmark} (5~attack scenarios, 200~probes) in
        which the middleware blocks 67~additional attacks and eliminates all
        PII leakage (300$\to$0~rows; \S\ref{sec:security-eval}).

  \item An \emph{agent-efficiency benchmark} (8~analytical tasks, 3~LLM
        models, up to 10~sequential agents) that reduces queries hitting the
        OLAP system by 74.5\% and eliminates all 95~schema-exploration
        queries (\S\ref{sec:agent-eval}).  An ablation study shows that
        removing the explore endpoint drops task success from 100\% to 12.5\%
        (\S\ref{sec:explore-ablation}), and cross-task transfer experiments
        demonstrate a 90\% query reduction when agents benefit from prior
        sessions on related tasks (\S\ref{sec:cross-task}).
\end{itemize}

\section{Background and Motivation}
\label{sec:background}

\subsection{Agentic Workload Characteristics}

Analysis of agent interactions with data systems on the
BIRD benchmark~\cite{li2024bird,liu2025sao} and multi-database tasks
identifies four distinguishing characteristics of agentic workloads:

\begin{description}[leftmargin=*,labelwidth=1.5cm]
  \item[Scale.]
    Agents generate orders-of-magnitude more queries than human analysts per
    task, driven by iterative exploration and trial-and-error solution
    formulation.  In the \sysname benchmark, baseline agents (without
    middleware) issue an average of 7.9~queries per analytical task, of which
    47.5\% are schema-exploration probes that never contribute to the final
    answer.

  \item[Heterogeneity.]
    Agent probes span a wide spectrum: coarse-grained metadata discovery
    (\texttt{system.columns}, \texttt{information\_schema}), partial query
    attempts that test column availability, and final analytical computation.
    Agents follow overlapping phases---metadata exploration, statistics
    gathering, and solution formulation---rather than a linear pipeline.

  \item[Redundancy.]
    Across sessions and agents, many probes access similar data or perform
    overlapping operations.  Distinct sub-plans across 50~parallel agent
    attempts comprise less than 10--20\% of total
    work, indicating 80--90\% redundancy.  Providing semantic hints reduced
    schema-exploration queries by 14--37\% depending on the exploration phase.

  \item[Steerability.]
    Unlike traditional query interfaces that return only results or errors,
    agents can consume and act on richer feedback---query suggestions,
    access-level annotations, constraint explanations, and semantic
    metadata---if the data system provides it.  The \sysname explore-endpoint
    ablation (\S\ref{sec:explore-ablation}) confirms this: agents with
    curated schema context succeed on 100\% of tasks; without it, they
    succeed on only 12.5\%.
\end{description}

These characteristics motivate a middleware layer that addresses redundancy
through proactive metadata exposure, leverages steerability through structured
feedback, governs agent resource consumption, and imposes security constraints
that understand agent intent rather than merely restricting column access.

\subsection{Limitations of Existing Approaches}
\label{sec:limitations}

\paragraph{Text-to-SQL systems.}
Systems like DIN-SQL~\cite{pourreza2024dinsql} focus on translating natural
language to SQL with high accuracy. However, they treat the data system as a
black box and do not address the security or efficiency dimensions of agentic
interaction---an agent using DIN-SQL still needs schema discovery and still
lacks intent-aware access control.

\paragraph{Traditional access control.}
RBAC~\cite{sandhu1996rbac} and fine-grained access control
systems~\cite{rizvi2004extending} enforce policies at the row, column, or
view level. These mechanisms cannot distinguish between query intents over the
same column.  For example, consider a \texttt{transactions} table with an
\texttt{amount\_cents} column classified as financial data.  Under RBAC, an
administrator must either grant access to the column (exposing individual
transaction amounts) or revoke it (preventing even aggregate revenue
analysis).  Intent-based access control resolves this by allowing
\texttt{SELECT category, AVG(amount\_cents) FROM transactions GROUP BY
category} (aggregate, low privacy risk) while blocking \texttt{SELECT
customer\_id, amount\_cents FROM transactions LIMIT 100} (row-level, high
privacy risk).

\paragraph{Query guardrails.}
LLM guardrail frameworks~\cite{rebedea2023nemo} operate at the prompt level,
filtering natural-language inputs before they reach the model. They do not
inspect the \emph{generated} SQL and cannot enforce data-level policies.
An agent that passes a benign natural-language prompt through a guardrail may
still generate SQL that exfiltrates PII, issues DDL commands, or floods the
system with expensive queries.  The guardrail layer and the data-access layer
are decoupled, leaving a gap that the middleware closes by inspecting every
query at the SQL AST level before it reaches the database.

\paragraph{Resource management.}
OLAP systems such as ClickHouse~\cite{clickhouse2024} offer per-user resource
governors---maximum memory, execution time, and thread limits.  These are
static limits configured at the user level, designed for environments with tens
of concurrent human analysts.  They cannot dynamically throttle individual
agents within a shared credential, enforce per-agent request budgets, or
provide backoff signals (\texttt{Retry-After} headers) that enable agents to
self-regulate.  The middleware layer addresses this gap by binding each agent
to a principal identity and enforcing sliding-window rate limits independently
of the OLAP engine's resource governors.

% ----- 3.1 Access Control -----
\subsection{Intent-Based Access Control}
\label{sec:access}

Traditional access control answers a binary question: \emph{can this principal
access this column?} \sysname asks a richer question: \emph{what does this
principal intend to do with this column, and is that intent permitted?}

\paragraph{Query intent classification.}
\sysname parses every incoming SQL statement into an AST using
\texttt{sqlglot}~\cite{sqlglot2023} and classifies the query into one of three
intent categories:

\begin{itemize}[leftmargin=*]
  \item \texttt{SELECT\_ROW}: Row-level data retrieval. The query returns
        individual records from sensitive columns without aggregation.
  \item \texttt{AGGREGATE}: Statistical computation. The query applies
        aggregate functions (\texttt{AVG}, \texttt{SUM}, \texttt{COUNT}, etc.)
        over sensitive columns, returning only summary statistics.
  \item \texttt{DDL/DML}: Schema modification or data mutation
        (\texttt{CREATE}, \texttt{INSERT}, \texttt{UPDATE}, \texttt{DELETE}).
\end{itemize}

\paragraph{Resource registry.}
Every column in the schema is annotated with a sensitivity classification
drawn from a controlled vocabulary:

\begin{itemize}[leftmargin=*]
  \item \texttt{PII} --- Personally identifiable information (\eg, names,
        emails, SSNs).
  \item \texttt{FINANCIAL} --- Revenue, salary, pricing data.
  \item \texttt{BUSINESS\_SECRET} --- Proprietary metrics, internal
        classifications.
  \item \texttt{SAFE} --- Non-sensitive columns (\eg, timestamps, categories).
\end{itemize}

\paragraph{Policy enforcement.}
The access control service composes intent classification with resource
sensitivity to enforce policies. For example, a default policy permits
\texttt{AGGREGATE} queries over \texttt{PII} columns (revealing
\texttt{AVG(salary)} but not individual salaries) while blocking
\texttt{SELECT\_ROW} access. Algorithm~\ref{alg:access} formalizes this logic.

\begin{algorithm}[t]
\caption{Intent-Based Access Control}
\label{alg:access}
\begin{algorithmic}[1]
\Require SQL query $q$, agent role $r$, resource registry $\mathcal{R}$, policy set $\mathcal{P}$
\Ensure \textsc{Allow} or \textsc{Deny} with explanation
\State $\mathit{ast} \gets \textsc{Parse}(q)$ \Comment{sqlglot AST}
\State $\mathit{intent} \gets \textsc{ClassifyIntent}(\mathit{ast})$
        \Comment{\texttt{SELECT\_ROW} | \texttt{AGGREGATE} | \texttt{DDL\_DML}}
\State $\mathit{cols} \gets \textsc{ExtractColumns}(\mathit{ast})$
\For{each column $c \in \mathit{cols}$}
    \State $\mathit{sens} \gets \mathcal{R}[c]$
        \Comment{PII | FINANCIAL | \ldots}
    \State $\mathit{decision} \gets \mathcal{P}.\textsc{Evaluate}(r, \mathit{intent}, \mathit{sens})$
    \If{$\mathit{decision} = \textsc{Deny}$}
        \State \Return $(\textsc{Deny},\;
               \text{``}\mathit{intent}\text{ over }\mathit{sens}\text{ column }c\text{ is blocked''})$
    \EndIf
\EndFor
\State \Return $(\textsc{Allow},\; \emptyset)$
\end{algorithmic}
\end{algorithm}

When a query is denied, \sysname returns a structured policy-violation response
rather than an opaque error. This leverages the \emph{steerability}
characteristic: agents receive actionable feedback explaining \emph{why} the
query was blocked and \emph{what} modifications would make it permissible (\eg,
``wrap column \texttt{salary} in an aggregate function'').

% ----- 3.2 Query Learning -----
\subsection{Session-Based Query Learning}
\label{sec:learning}

Agentic workloads exhibit high redundancy: agents tackling similar analytical
objectives issue overlapping query sequences.  \sysname captures this
redundancy through session-based query learning.

\paragraph{Session lifecycle.}
An agent opens a session by providing a natural-language description of its
analytical objective (\eg, ``Analyze Q3 revenue trends by region'').  During
the session, the agent executes queries and provides relevance feedback---a
boolean signal indicating whether each query's result was useful, optionally
accompanied by a free-text note.  When the session closes, \sysname computes
an embedding of the session description and persists the session (description
embedding, query--feedback tuples) in a vector store.

\paragraph{Knowledge transfer.}
When a new agent opens a session, \sysname performs a top-$K$ similarity
search over stored session embeddings and returns the most relevant past
queries.  Only queries with positive feedback are surfaced, ensuring that
the knowledge base is curated rather than a raw dump of all prior queries.
Negatively-reviewed queries that were blocked by access control are also
returned as warnings, helping agents avoid known policy violations.

\paragraph{Cross-task transfer.}
Because retrieval is embedding-based rather than keyword-based, knowledge
transfers across related but distinct tasks.  An agent solving
``revenue by product category'' stores queries against the \texttt{revenue\_daily}
table.  A subsequent agent solving ``refund rate by category''---a different
analytical question over the same table---receives those queries as relevant
context.  In experiments (\S\ref{sec:cross-task}), this cross-task transfer
reduces queries by 90\% compared to baseline.

\paragraph{Saturation dynamics.}
Session learning exhibits a saturation curve: the marginal value of each
additional stored session diminishes as the embedding store fills.  In
practice (\S\ref{sec:saturation}), similar-query retrieval saturates at
$K{=}6$ queries after approximately 7~sequential agents, after which
additional sessions add no new information.  Token consumption stabilizes
at this point as well.

% ----- 3.3 Explore Endpoint -----
\subsection{Curated Explore Endpoint}
\label{sec:explore}

The explore endpoint directly addresses the \emph{redundancy} characteristic
by proactively surfacing the metadata that agents would otherwise discover
through iterative probing.  For each table accessible to the agent, the
endpoint returns:

\begin{itemize}[leftmargin=*]
  \item \textbf{Table descriptions}: Human-authored summaries of table purpose
        and contents.
  \item \textbf{Column semantics}: Data types, natural-language descriptions,
        sensitivity classifications, and \emph{access-level annotations}
        (\eg, \texttt{full}, \texttt{aggregation\_only}, \texttt{denied}).
  \item \textbf{Statistics}: Row counts and basic column statistics
        (min, max, average) for accessible columns.
  \item \textbf{Sample queries}: Curated example queries demonstrating
        policy-compliant access patterns.
  \item \textbf{Join recommendations}: Pre-computed join paths between tables
        with foreign-key annotations.
\end{itemize}

The explore endpoint is \emph{access-controlled}: tables the agent has no
access to are omitted entirely, and restricted columns are annotated with
their access level rather than hidden.  This design serves dual purpose:
\emph{schema discovery} (what tables and columns exist) and \emph{policy
guidance} (how to query within access constraints).  For example, a column
marked \texttt{aggregation\_only} teaches the agent to use
\texttt{AVG(column)} rather than \texttt{SELECT column}---turning access
control from a brick wall into a navigable constraint.  The ablation study
in \S\ref{sec:explore-ablation} confirms this: without explore, even GPT-4o
abandons tasks after hitting policy blocks, interpreting them as hard denials
rather than formulation guidance.

% ----- 3.4 Resource Governance -----
\subsection{Agent Resource Governance}
\label{sec:governance}

The middleware's position as an interposition layer enables resource governance
that the OLAP engine cannot provide natively.  Each agent is bound to a
principal identity---regardless of how many sessions or connections it
opens---and subjected to three governance mechanisms:

\begin{itemize}[leftmargin=*]
  \item \textbf{Per-agent rate limiting}: A sliding-window rate limiter
        tracks each agent's query frequency independently.  When the limit
        is exceeded, the middleware returns HTTP~429 with a
        \texttt{Retry-After} header, enabling well-behaved agents to back
        off automatically.
  \item \textbf{Audit logging}: Every query is logged with the agent's
        identity, the tables and columns accessed, the access-control
        decision, execution time, and rows read.  This enables
        retrospective analysis of agent behavior patterns.
  \item \textbf{Query interception}: Because every query passes through
        the middleware before reaching the OLAP engine, dangerous functions
        (\texttt{url()}, \texttt{file()}, \texttt{remote()}) and system
        commands (\texttt{SYSTEM SHUTDOWN}, \texttt{KILL QUERY}) are
        blocked before they consume cluster resources.
\end{itemize}

\section{Implementation}
\label{sec:implementation}

The \sysname implementation, benchmark harness, and all experimental scripts
are available at
\texttt{https://github.com/temp-noob/soa-harness}.
Table~\ref{tab:stack} summarizes the technology stack.

\begin{table}[t]
  \centering
  \caption{Technology stack.}
  \label{tab:stack}
  \begin{tabular}{@{}ll@{}}
    \toprule
    \textbf{Component} & \textbf{Technology} \\
    \midrule
    OLAP Engine         & ClickHouse 24.8 (2-shard, 2-replica) \\
    HTTP Proxy          & FastAPI + Uvicorn (async) \\
    SQL Parsing         & sqlglot \\
    Embedding Store     & ChromaDB (v1.0, HTTP API) \\
    Embedding Model     & Configurable (OpenAI-compatible API) \\
    Agent Models        & GPT-4o, GPT-4o-mini, Qwen-3.5 \\
    \bottomrule
  \end{tabular}
\end{table}

\paragraph{Deployment model.}
The middleware deploys as a containerized HTTP server that accepts
OLAP-compatible HTTP protocol requests.  In the evaluation, it proxies
ClickHouse's HTTP interface; agents configured with
\texttt{clickhouse\_connect} or any ClickHouse HTTP client require only a
change in endpoint URL.  The middleware, OLAP cluster, and vector store are
orchestrated via Docker Compose.

\paragraph{SQL parsing and intent classification.}
\sysname uses \texttt{sqlglot}~\cite{sqlglot2023} to parse incoming SQL into a
full AST.  The intent classifier detects the presence of aggregate functions
(\texttt{COUNT}, \texttt{SUM}, \texttt{AVG}, etc.) and \texttt{GROUP BY}
clauses to distinguish \texttt{AGGREGATE} intent from \texttt{SELECT\_ROW}.
DDL and DML statements are identified by keyword prefix.  Dangerous
table-valued functions (\texttt{url()}, \texttt{file()}, \texttt{remote()})
are detected and blocked before AST analysis.

\paragraph{Session embedding.}
Session descriptions are embedded via a configurable OpenAI-compatible
embedding API (set via the \texttt{EMBEDDING\_API\_URL} environment variable).
When no API is configured, a lightweight local trigram-hash function produces
64-dimensional unit-norm vectors as a fallback.  Embeddings are stored in
ChromaDB via its REST API, with metadata including the agent~ID, session
timestamp, query--feedback tuples (as JSON), and query count.  Retrieval uses
cosine similarity with $K{=}6$.

\paragraph{Resource registry.}
The resource registry is a YAML configuration file that annotates every table
and column in the schema with a sensitivity classification (\texttt{PII},
\texttt{FINANCIAL}, \texttt{BUSINESS\_SECRET}, \texttt{SYSTEM}, or
\texttt{SAFE}) and a human-readable description.  Access policies are
defined in a separate YAML file that maps agent roles to permitted
(intent, sensitivity) combinations.  Both files are loaded at startup and
can be updated without restarting the middleware by calling the
\texttt{/explore/refresh} endpoint.

\section{Evaluation}
\label{sec:eval}

\sysname is evaluated along two dimensions: security posture
(\S\ref{sec:security-eval}) and agent efficiency
(\S\ref{sec:agent-eval}).  All experiments use a distributed
ClickHouse~\cite{clickhouse2024} cluster (2~shards $\times$ 2~replicas) as
the OLAP backend, populated with a synthetic e-commerce analytics
dataset (5~tables: customers, transactions, products, revenue\_daily,
agent\_audit\_log; 200K+ transactions across 5K~customers and
500~products).

% ----- 5.1 Security -----
\subsection{Security Benchmark}
\label{sec:security-eval}

\paragraph{Methodology.}
Four attack scenarios span common threat vectors for agentic database access,
totaling 201~adversarial probes executed via the same
\texttt{clickhouse\_connect} client that agents use:

\begin{enumerate}[leftmargin=*]
  \item \textbf{Resource exhaustion} (13~probes) --- Cartesian joins, memory
        bombs (\texttt{arrayJoin(range(N))}), regexp full scans, unbounded
        high-cardinality \texttt{GROUP BY}, and nested subquery explosions.
  \item \textbf{Concurrent flooding} (150~probes) --- 50~concurrent agents,
        3~rounds, mixing analytical scans with exploration probes.
  \item \textbf{Data exfiltration} (13~probes) --- Schema reconnaissance,
        direct PII access, aggregation-based inference, and system-catalog
        snooping.
  \item \textbf{Query injection} (25~probes) --- Classic DDL/DML attacks
        (caught by \texttt{readonly=1}) \emph{plus} advanced probes that
        bypass \texttt{readonly}: table-function abuse (\texttt{url()},
        \texttt{remote()}, \texttt{s3()}, \texttt{mysql()}), system-table
        information leakage, and multi-statement injection.
\end{enumerate}

\paragraph{Results.}
Table~\ref{tab:security} presents the results.  The middleware blocks
79~additional probes compared to unprotected ClickHouse, with the strongest
gains on resource exhaustion, query injection, and data exfiltration.

\begin{table}[t]
  \centering
  \caption{%
    Security benchmark: 4~attack scenarios, 201~probes total.
    Block~\% indicates fraction of probes denied or rate-limited.

\paragraph{Resource exhaustion.}
The middleware's AST-level detection catches cartesian joins, memory bombs
(\texttt{arrayJoin(range(N))} with $N > 10{,}000$), regexp full scans on
large tables, and unbounded high-cardinality \texttt{GROUP BY}---blocking
11 of 13~probes (85\%) compared to only 3 in baseline.  These queries are
rejected \emph{before} reaching the OLAP engine, eliminating the resource
consumption entirely.

\paragraph{Data exfiltration.}
PII rows leaked drops from 300 to~0 across all exfiltration probes.  The
intent-based access control identifies row-level \texttt{SELECT} queries
over PII columns and blocks them while permitting aggregate computations
over the same columns.  System-catalog snooping
(\texttt{system.query\_log}, \texttt{system.processes}) is also blocked.

\paragraph{Query injection.}
The middleware achieves a perfect 100\% block rate (25/25) versus 84\% for
baseline.  ClickHouse's \texttt{readonly=1} mode blocks classic DDL/DML
attacks, but 4~probes exploit vectors that \texttt{readonly} does
\emph{not} catch: table-function abuse (\texttt{url()}, \texttt{s3()},
\texttt{remote()}, \texttt{mysql()}) for data exfiltration and server
probing, system-table information leakage, and multi-statement injection.
The middleware's AST-level inspection blocks all of these before they reach
the database.

\paragraph{Concurrent flooding.}
The middleware's per-agent rate limiting blocks 40\% of flooding probes via
sliding-window enforcement with \texttt{Retry-After} headers.  The
incomplete coverage reflects the difficulty of distinguishing legitimate
high-throughput analytical workloads from adversarial flooding without
false positives.

% ----- 5.5 Saturation -----
\subsection{Session Learning Saturation}
\label{sec:saturation}

Ten sequential agents run on three tasks with feedback enabled
(Figure~\ref{fig:saturation}).  Similar-query retrieval saturates at $K{=}6$
(the top-$K$ cap) after approximately 7~agents for the first task
(\texttt{revenue\_by\_category}).  Subsequent tasks
(\texttt{category\_refund\_rate}, \texttt{product\_category\_aov}) saturate
immediately at Run~1 due to cross-task transfer from earlier tasks.

Simple tasks converge to 1~query per agent; the complex task
(\texttt{product\_category\_aov}, requiring a policy-restricted JOIN)
plateaus at 3~queries---an irreducible floor imposed by access control, not
by schema discovery.  Token consumption stabilizes once similar-query
retrieval saturates, confirming that the session store reaches a steady state.

The rapid plateau is partly an artifact of the benchmark's compact schema
(5~tables, 200K~rows).  On a production-scale warehouse with hundreds of
tables, deeper join hierarchies, and more diverse analytical patterns, the
saturation curve would likely be more gradual---with session learning
providing incremental value over a longer horizon as agents encounter a
wider variety of query patterns before the embedding store covers the
schema surface area.

% ----- 5.7 Feedback -----
\subsection{Feedback Efficacy}
\label{sec:feedback}

To isolate the value of agent feedback in session learning, the benchmark
is run with feedback enabled versus disabled
(\texttt{-{}-no-feedback}).  Without feedback, sessions are persisted but
stored queries have no relevance signal; the retrieval logic only surfaces
queries with positive feedback, so no-feedback sessions are effectively
invisible to future agents.

With feedback, match rate improves from 29.2\% to 37.5\% (+8.3~pp).
The largest gain is on \texttt{product\_category\_aov}, where
feedback-curated queries help Run~2 and Run~3 agents avoid the
policy-blocked formulations that Run~1 encountered.  This confirms that
feedback is the curation mechanism that makes session learning
effective---without it, the session store accumulates queries but cannot
distinguish useful patterns from dead ends.

\section{Discussion}
\label{sec:discussion}

\subsection{Limitations}

\paragraph{Model capability dependence.}
The middleware amplifies model capability rather than leveling it
(\S\ref{sec:model-comparison}).  Qwen-3.5 achieves only 25\% accuracy
despite full middleware support, consuming 10$\times$ more tokens than
GPT-4o due to verbose retry loops when policy blocks are encountered.
This is partially architectural---the middleware could pre-validate likely
query patterns and warn agents proactively---and partially a function of
model capability.  As the model quality dimension shows, the middleware's
value increases with agent capability: stronger models interpret
\texttt{aggregation\_only} annotations and formulate compliant SQL on the
first attempt.

\paragraph{Static resource registry.}
The current resource registry requires manual column classification via
YAML configuration.  In production, this annotation burden could be reduced
through automated PII-detection classifiers, schema-metadata propagation
from enterprise data catalogs (\eg, Apache Atlas, Collibra), or inference
from column names and sample values.  The middleware's modular design
accommodates such extensions without architectural changes.

\paragraph{Benchmark representativeness.}
The evaluation uses a compact synthetic dataset (5~tables, 200K~rows).
Production OLAP warehouses have hundreds of tables, deeper join hierarchies,
and more diverse query patterns.  The rapid session-learning saturation
observed in \S\ref{sec:saturation} would likely be more gradual on such
systems, with the middleware providing value over a longer horizon.

\paragraph{Side-channel timing attacks.}
The middleware does not currently mitigate timing-based inference attacks.
An adversary can progressively narrow \texttt{WHERE} clauses in aggregate
queries (\eg, \texttt{SELECT COUNT(*) WHERE credit\_score > 700}, then
\texttt{> 750}, then \texttt{> 775}) to infer individual values through
latency differences.  The middleware's position as an interposition layer
makes it the natural place to add such defenses, but the privacy--utility
trade-offs require careful study.

\subsection{Future Work}

\begin{itemize}[leftmargin=*]
  \item \textbf{Query cost estimation.}
        The middleware already parses every query into a full SQL AST for
        intent classification.  A natural extension is to estimate scan
        cost from the AST---row counts from table statistics, join
        cardinality from key distributions---and reject queries with
        estimated costs above a configurable threshold \emph{before}
        execution.  Combined with the existing resource exhaustion
        pattern detection, this would close the remaining gap in
        resource protection.

  \item \textbf{Adaptive policy feedback.}
        Tailoring the verbosity and specificity of policy-violation
        explanations to the agent's demonstrated capability level.
        A weak model might receive in-context examples of compliant
        query rewrites, while a strong model receives only the column
        annotation.

  \item \textbf{Inter-probe optimization.}
        Multi-query optimization for agentic
        workloads~\cite{liu2025sao,sellis1988mpo} is a promising direction.
        The session learning
        component could be extended to precompute and cache common
        sub-expressions across agent sessions.

  \item \textbf{Side-channel defenses.}
        Integrating differential privacy mechanisms (calibrated noise on
        aggregate results) or minimum group-size thresholds into the
        access control pipeline.  The middleware already inspects every
        query at the AST level, making it the natural enforcement point
        for such defenses.
\end{itemize}

\section{Related Work}
\label{sec:related}

\paragraph{Agent-first data systems.}
Liu \etal~\cite{liu2025sao} provide the foundational analysis of agentic
workloads and propose an agent-first data systems architecture addressing
query interfaces, probe optimization, indexing, and storage.  This work
takes inspiration from their characterization of agentic workloads and
develops a benchmark harness and middleware prototype to explore how such
an architecture might help in practice.  The evaluation carves out
measurable dimensions---security posture, query efficiency, explore
ablation, cross-task transfer, session saturation, model scaling, and
feedback efficacy (\S\ref{sec:eval})---to assess where middleware-level
interventions provide the most value without modifying the underlying
OLAP engine.

\paragraph{Text-to-SQL.}
The text-to-SQL literature~\cite{yu2018spider,li2024bird,pourreza2024dinsql}
focuses on the accuracy of translating natural language to SQL.  These systems
are complementary to \sysname: they generate queries that then flow through
the middleware for access control and learning.  \sysname addresses the
orthogonal challenges of security and efficiency that arise \emph{after}
query generation---a text-to-SQL system producing perfect SQL still needs
intent-based access control and schema context.

\paragraph{Database access control.}
RBAC~\cite{sandhu1996rbac} and query-rewriting
approaches~\cite{rizvi2004extending} enforce access at the row or column
level.  \sysname extends this with intent-based classification,
distinguishing aggregation from row-level access over the same column.
The key difference is \emph{granularity of intent}: traditional systems
answer ``can this principal access this column?'' while \sysname answers
``what does this principal intend to do with this column, and is that
intent permitted?''  This distinction is critical for agentic workloads
where the same column legitimately appears in both aggregation and
row-level queries.

\paragraph{LLM guardrails.}
NeMo Guardrails~\cite{rebedea2023nemo} and similar frameworks enforce safety
constraints at the prompt level, filtering natural-language inputs before
they reach the model.  These operate upstream of query generation and cannot
inspect the SQL that agents produce.  \sysname operates downstream,
inspecting and enforcing policies on the actual queries sent to the
database.  The two approaches are complementary: guardrails constrain what
agents \emph{ask for}, while \sysname constrains what agents
\emph{actually access}.  The security benchmark demonstrates that this
downstream inspection catches vectors that upstream guardrails miss---
table-function abuse (\texttt{url()}, \texttt{s3()}), system-table
information leakage, and multi-statement injection.

\paragraph{Semantic caching.}
Semantic caching~\cite{dar1996semantic} stores query results indexed by
their semantic content, enabling cache hits for queries with overlapping
predicates.  \sysname's session-based learning is related but distinct:
rather than caching results, it caches \emph{query patterns and relevance
signals}, enabling knowledge transfer at the strategic rather than
operational level.  The cross-task transfer experiment
(\S\ref{sec:cross-task}) demonstrates that this approach works across
different analytical questions sharing underlying tables.

\paragraph{Data discovery and metadata management.}
Systems like Aurum~\cite{fernandez2018aurum} help users discover relevant
datasets and relationships.  \sysname's explore endpoint serves a similar
purpose for agents, but adds two critical features: \emph{access-level
annotations} that teach agents how to formulate policy-compliant queries,
and \emph{access filtering} that omits tables and columns the agent
cannot use, reducing both the attack surface and the agent's schema
confusion.

\paragraph{Agent tool use and reasoning.}
ReAct~\cite{yao2023react} and Toolformer~\cite{schick2024toolformer}
establish the paradigm of LLM agents interacting with external tools
through iterative reasoning and action.  \sysname provides the data-system
side of this interaction---structured interfaces (explore, session
learning) that are designed for programmatic consumption by reasoning
agents rather than human analysts.

\section{Conclusion}
\label{sec:conclusion}

As LLM agents become primary consumers of analytical data systems, the
interface between agents and databases requires fundamental rethinking.
This work presented \sysname, a transparent middleware layer that introduces
intent-based access control, session-based query learning, curated schema
discovery, and agent resource governance for OLAP systems.

The evaluation on a distributed ClickHouse cluster demonstrates that
middleware-level interventions can significantly improve both the security
and efficiency of agentic database access:

\begin{itemize}[leftmargin=*]
  \item The security benchmark shows a 4$\times$ increase in attack
        blocking (13\%$\to$53\% of 201~probes), with PII leakage eliminated
        entirely, resource exhaustion detection at 85\%, and query injection
        at 100\%---including vectors that ClickHouse's \texttt{readonly}
        mode does not catch.

  \item The agent efficiency benchmark reduces OLAP queries by 74.5\%
        (200$\to$51) and eliminates all 95~schema-exploration queries.
        The explore endpoint is the foundational enabler: without it,
        task success drops from 100\% to 12.5\%.

  \item Cross-task knowledge transfer reduces queries by 90\% for
        semantically related tasks, and session learning saturates at
        $K{=}6$ similar queries after approximately 7~agents.

  \item Model capability determines how effectively agents leverage the
        middleware: GPT-4o achieves 62.5\% accuracy in 1.1~queries,
        GPT-4o-mini 45.8\% in 3.5~queries, and Qwen-3.5 25\% in
        5.7~queries---confirming that the middleware amplifies model
        capability rather than leveling it.
\end{itemize}

These results validate the agent-first design
principles~\cite{liu2025sao} and demonstrate that substantial progress
is achievable at the middleware layer without modifying the underlying
OLAP engine.  The middleware architecture---transparent proxy, AST-level
inspection, embedding-based session memory---is database-agnostic by
design and extensible to other OLAP backends.
